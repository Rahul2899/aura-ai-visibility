# Phase 4 — AWS EC2 Production Deployment

## Goal
Deploy the full stack (FastAPI + Next.js + Postgres) on AWS EC2 t2.micro using Docker Compose.
Learn: SSH, Docker on a real server, Nginx reverse proxy, how to debug when things break remotely.

## What Was Built
- `Dockerfile` — Python 3.12-slim, installs requirements, runs `alembic upgrade head` then uvicorn
- `web/Dockerfile` — Node 22 Alpine, builds Next.js, serves with `next start`
- `nginx/nginx.conf` — Nginx reverse proxy, port 80 → Next.js on port 3000
- `docker-compose.yml` — All 4 services: db (postgres:16), app (FastAPI:8000), web (Next.js:3000), nginx (:80)

## Architecture Decision: Why API on 8000 Directly (Not Through Nginx)

Next.js has two runtimes: server-side rendering (runs inside Docker container) and browser (runs on user's laptop).

If nginx proxied `/api/` → FastAPI, browser calls would work, but server-side `fetch()` calls would fail because the web container can't reach nginx from inside Docker. Two different URL systems.

Simpler solution: expose FastAPI on port 8000 directly. Both SSR and browser use `NEXT_PUBLIC_API_URL=http://EC2_IP:8000`. One URL, always works.

Tradeoff: port 8000 must be open in EC2 security group. Acceptable for this stage.

## AI vs Non-AI Decisions
- Dockerfile CMD `alembic upgrade head && uvicorn ...` — not AI, shell command sequence
- healthcheck on postgres — not AI, deterministic ping
- `restart: unless-stopped` — not AI, Docker restart policy
- NEXT_PUBLIC_API_URL config — not AI, environment variable routing

## Critical: NEXT_PUBLIC_API_URL Must Be Set Before Build

Next.js bakes `NEXT_PUBLIC_*` vars at build time (when `npm run build` runs). If you `docker compose build` without setting it, the production app will have the default value burned in and can't be changed without rebuilding.

**Correct order:**
1. Set `NEXT_PUBLIC_API_URL=http://YOUR_EC2_IP:8000` in docker-compose.yml (or pass as build arg)
2. Run `docker compose build`
3. Run `docker compose up -d`

## EC2 Setup Steps

**1. Launch EC2 instance**
- Go to AWS Console → EC2 → Launch Instance
- AMI: Ubuntu 24.04 LTS
- Instance type: t2.micro (free tier)
- Key pair: create new, download .pem file
- Security group: allow SSH (22), HTTP (80), custom TCP 8000 from anywhere

**2. Assign Elastic IP**
- EC2 → Elastic IPs → Allocate → Associate to instance
- This gives you a static IP that doesn't change on restart

**3. Connect via SSH**
```bash
chmod 400 your-key.pem
ssh -i your-key.pem ubuntu@YOUR_ELASTIC_IP
```

**4. Install Docker on the server**
```bash
sudo apt update
sudo apt install -y docker.io docker-compose-v2
sudo usermod -aG docker ubuntu
# Log out and back in for group change to take effect
exit
ssh -i your-key.pem ubuntu@YOUR_ELASTIC_IP
```

**5. Copy project to server**
```bash
# From your laptop:
scp -i your-key.pem -r /Users/rahul2202/Downloads/Peecclone ubuntu@YOUR_ELASTIC_IP:~/peecclone
scp -i your-key.pem /Users/rahul2202/Downloads/Peecclone/.env ubuntu@YOUR_ELASTIC_IP:~/peecclone/.env
```

**6. Edit NEXT_PUBLIC_API_URL before building**
```bash
# On the server:
cd ~/peecclone
# Edit docker-compose.yml, set NEXT_PUBLIC_API_URL to your Elastic IP
nano docker-compose.yml
# Change: NEXT_PUBLIC_API_URL: http://YOUR_ELASTIC_IP:8000
```

**7. Build and start**
```bash
docker compose build
docker compose up -d
```

**8. Verify**
```bash
docker compose ps               # all 4 services should show "running"
docker compose logs app         # check FastAPI started, migrations ran
curl http://localhost:8000/brands  # API responds
curl http://localhost:80        # Nginx responds with Next.js HTML
```

**Access in browser:**
- Frontend: `http://YOUR_ELASTIC_IP`
- API docs: `http://YOUR_ELASTIC_IP:8000/docs`

## Debugging Reference

| Symptom | First check | Command |
|---------|-------------|---------|
| App not starting | Logs | `docker compose logs app` |
| DB connection refused | Is postgres healthy? | `docker compose ps db` |
| Migrations failed | App logs | `docker compose logs app --tail 50` |
| Port 8000 unreachable | Security group | AWS Console → EC2 → Security Groups |
| Out of memory (1GB RAM) | Usage | `free -h && docker stats` |
| Nginx 502 | Is web container up? | `docker compose ps web` |
| Next.js build failed | Build logs | `docker compose logs web` |
| Disk full from old images | Prune | `docker system prune -f` |

## What I Learned
- NEXT_PUBLIC_* env vars are baked at build time, not runtime. This is a common Next.js gotcha.
- Docker healthcheck on postgres prevents app starting before DB is ready. Without this, alembic migrations fail on startup.
- `restart: unless-stopped` means containers auto-recover if they crash — except if you explicitly `docker stop` them.
- SSH + Docker Compose is the minimal production workflow. Understanding this directly = understanding how Heroku/Render/Railway actually work under the hood.
- t2.micro has 1GB RAM. Next.js build (`npm run build`) uses ~600MB alone. Might fail on first run — if so, add swap: `sudo fallocate -l 1G /swapfile && sudo chmod 600 /swapfile && sudo mkswap /swapfile && sudo swapon /swapfile`
